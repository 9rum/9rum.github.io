Docker를 이용해 test-tlb 실행하기

April 14, 2022

2022   ·   container   docker   linux   os     ·   container  

블로그 시작하기 좋은 날입니다.

첫 글로 뭘 쓸까 고민하다 친구가 잘 하는 것부터 써보라고 해서, 잘 하는 게 없는 저는 대신 해봤던 걸 쓰기로 했습니다.

이 글은 제가 OS 수업에서 Docker를 사용해 과제를 해결했던 방법을 소개합니다.
테스트가 안돌아가서 고민하는 분들이 도움을 얻어가면 좋겠습니다.

0. test-tlb

학부생 시절 OS 수업에서는 이맘때쯤 test-tlb라는 과제가 나왔었습니다.
test-tlb를 사용해 memory latency를 측정하고 거꾸로 실험 환경의 TLB(Translation Lookaside Buffer) 구조를 예측해 보는 과제였죠.

test-tlb는 Linus Torvalds가 만든, TLB에 의한 memory latency 감소 효과를 실험해 볼 수 있는 비교적 간단한 툴입니다.
GitHub에 공개돼 있는 소스를 다운 받아 make로 빌드하면 바로 실행 파일을 얻을 수 있습니다.
그리고 test-tlb는 Linux를 대상 운영체제로 작성되어 있습니다.

1. 운영체제와 실행 파일

요새는 연구하느라 거의 Linux를 써서 잘 안쓰지만, 당시 저는 맥북을 막 구매해서 맥북을 1초라도 더 쓰기 위해 재밌게 개발을 했던 것 같습니다.

Linux, Darwin, Windows와 그 외 많은 커널들은 각자의 목적과 방향성이 있고 서로 다른 장단점이 있지만, 서로 호환되지는 않습니다.

여기서 서로 호환되지 않는다는 것은 소프트웨어의 실행이 서로 호환되지 않는다는 뜻입니다.
즉, 임의의 운영체제에서 빌드한 응용 프로그램이 다른 운영체제에서는 실행될 수 없거나, 실행 결과가 항상 같을 것을 보장할 수 없다는 것이죠.

이는 (1) 각 운영체제가 서로 다른 실행 파일 포맷을 사용하고 (2) 컴파일러가 하나의 운영체제와 아키텍처에 맞는 하나의 실행 파일만을 생성하기 때문입니다.

다음 실행 결과는 test-tlb가 x86_64 Linux에서 실행될 수 있음을 나타냅니다.

  $ make
  gcc -g -Wall -O test-tlb.c -o test-tlb -lm
  $ file test-tlb
  test-tlb: ELF 64-bit LSB pie executable, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, BuildID[sha1]=b5ef04fe820566ec020c5a38238c4e6eae6cacac, for GNU/Linux 4.4.0, with debug_info, not stripped

각 운영체제와 아키텍처에 맞는 실행 파일을 만들기 위해 개발자가 매번 다른 코드를 개발하는 것은 상당히 비효율적이기 때문에 이런 문제를 해결하기 위해 JVM이나 cross compiler 등의 해결책들이 제시됐지만, 아쉽게도 test-tlb는 GCC를 사용합니다.
그리고 저는 macOS를 사용하고 있었습니다.

2. Docker

그래서 저는 Docker를 사용하기로 했습니다.

Virtual Box, VMWare 등의 가상머신을 사용할 수도 있지만, 설치와 부팅이 오래 걸린다는 단점이 있고 Apple Silicon 칩셋이 들어간 M1 Mac의 경우엔 아직 지원이 안됩니다.

반면 Docker는 수초 내로 실행되고 설치와 제거를 명령어 몇 줄로 수행할 수 있습니다.

왜 이런 차이가 생긴 걸까요? VM과 Docker의 동작 방식을 살펴보면 그 이유를 알 수 있습니다.

가상머신은 Hypervisor를 이용해 하드웨어를 가상화하고 그 위에 Guest OS를 실행해서 운영체제를 격리하는 방식으로 동작합니다.
하드웨어를 가상화하기 때문에 리소스 스케줄링이 복잡해지고 Guest OS를 부팅하기 때문에 Docker에 비해 느립니다.

반면 Docker는 Host OS 위에 Docker Engine이라는 데몬 프로세스가 실행되고 cgroup, namespace, chroot를 이용해 프로세스를 격리하는 방식으로 동작합니다.
별도의 가상화 없이 호스트 커널을 공유하는 프로세스만 생성하기 때문에 가상머신보다 빠릅니다. 반면 가상머신은 하드웨어를 가상화하기 때문에 interference 측면에서는 Docker보다 낫습니다.

3. Docker 설치하기

Linux 머신이 있다면 native로 실행하면 되고 Windows 머신이 있다면 WSL 2를 사용하는 편이 나을 수도 있습니다.
하지만 macOS에서는 여러모로 Docker를 사용하는 편이 낫고 내가 나중에 다시 보려고 Docker 사용법을 알아두면 활용할 곳이 많습니다.

  • macOS

    macOS에는 de facto 패키지 관리자인 Homebrew가 있습니다.
    패키지 관리자를 사용하는 편이 패키지 관리 측면에서 깔끔하고 효율적이기 때문에 Homebrew를 통해 설치하는 것을 추천합니다.
    Homebrew가 없다면 설치해 줍니다.

    $ /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

    설치하는 데 시간이 좀 걸릴 수도 있습니다. 커피 한 잔 내려서 한 모금 마시고 Docker를 설치해 줍니다.

    $ brew install --cask docker

  • Linux

    Linux는 배포판이 참 많습니다. Debian, RHEL, OpenSUSE, Arch Linux 등 배포판 계열마다 패키지 관리자도 상이합니다.
    하지만 학생들은 Debian 계열의 Ubuntu를 제일 많이 쓰니까 apt를 이용한 설치만 우선 적어두고 후에 추가하도록 하겠습니다.
    우선 HTTPS로 repository를 받아올 수 있도록 사전 설치를 해주고 Docker repository의 GPG key를 등록합니다.

    $ sudo apt update
$ sudo apt install ca-certificates curl gnupg lsb-release
$ curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg

    Docker의 stable repository를 추가합니다. nightly나 test 버전도 있지만 명확한 이유가 없다면 굳이 사용할 필요는 없습니다.

    $ echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/docker-archive-keyring.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

    이제 Docker repository가 추가되었으니 Docker를 설치해 줍니다.

    $ sudo apt update
$ sudo apt install docker-ce docker-ce-cli containerd.io

    잘 켜졌는데 매번 sudo를 붙여서 Docker Engine을 켜주는 건 귀찮습니다. sudo 없이 docker 명령을 실행하고 부팅 시 Docker Engine을 켜주도록 합니다.

    $ sudo usermod -aG docker $USER
$ sudo systemctl enable docker.service
$ sudo systemctl enable containerd.service

    이제 Dockerfile 작성해서 실험만 하면 됩니다.

4. Dockerfile 만들고 실행하기

Docker는 컨테이너라는 단위로 프로세스를 격리하고 관리합니다.
각 컨테이너는 Docker Image라는 리소스와 실행파일 묶음을 실행한 결과이고 개발자는 Dockerfile이라는 DSL(Domain Specific Language)과 docker 명령을 통해 Docker Image를 생성하고 이를 실행하여 컨테이너를 띄울 수 있습니다.

test-tlb를 실행하는 Docker Image를 만드려면 다음과 같은 Dockerfile을 작성하면 됩니다.

작성한 Dockerfile로 test-tlb라는 이름의 Docker Image를 빌드합니다.

  $ docker build -t test-tlb .

Docker Image가 잘 만들어졌는지 확인하려면 docker images를 사용하면 됩니다.

  $ docker images
  REPOSITORY   TAG      IMAGE ID       CREATED         SIZE
  test-tlb     latest   17a556141e8a   5 minutes ago   353MB
  ubuntu       20.04    825d55fb6340   9 days ago      72.8MB

이제 컨테이너를 띄워봅시다. 컨테이너 실행 명령은 docker run입니다.
결과를 파일로 저장하고 싶으면 redirection(>)을 사용하면 됩니다.

  $ docker run test-tlb > result

이제 Dockerfile의 마지막 줄만 make run 대신 map size와 stride를 바꿔가며 실행하면 됩니다.
